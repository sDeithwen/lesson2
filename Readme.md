### Практическая работа №4 «Первый Docker Compose: от одного контейнера к стеку»

Цель:
Научиться запускать многоконтейнерное приложение с помощью Docker
Compose, понимать механизмы работы сетей, томов, переменных окружения и
healthcheck.

Приложения:
- producer – Flask-приложение, которое:
	- читает файл /data/data.txt (bind‑mount) и раз в 5 секунд выводит его содержимое в консоль;
	- имеет эндпоинты: /health (возвращает "OK") и /data (возвращает содержимое файла);
	- при старте делает паузу 5 секунд, имитируя долгий запуск.
- consumer – скрипт, который раз в 5 секунд делает GET-запрос к http://producer:5000/data и выводит полученный текст; 
- при недоступности producer пишет в консоль ошибку.
#### Что нужно сделать
Нужно описать стек из двух сервисов в docker-compose.yml,
используя готовые образы из Docker Hub, настроить bind‑mount для файла
на хосте, добавить healthcheck и использовать переменные окружения через
.env. После этого запустить стек и убедиться, что оба контейнера работают и обмениваются данными.
#### Обязательные требования
- Использовать готовые образы с Docker Hub - sdeithwen/producer:latest и sdeithwen/consumer:latest.
- В docker-compose.yml сервис producer должен:
	- монтировать локальный файл data.txt в контейнер по пути /data/data.txt (bind‑mount);
- В docker-compose.yml сервис consumer должен:
	- иметь переменную окружения PRODUCER_URL=http://producer:5000;
	- зависеть от producer через depends_on (простая зависимость).
- Вся работа с Compose выполняется через команду docker-compose up, ручной запуск контейнеров через docker run не используется.
- Названия контейнеров consumer и producer
#### Базовая сдача
Для базовой сдачи должно быть выполнено следующее:
- Стек запускается без ошибок (docker-compose up).
- Оба контейнера работают (docker-compose ps).
- В логах producer (docker-compose logs producer) через 5 секунд после старта появляется вывод содержимого data.txt и значение переменной GREETING.
- В логах consumer (docker-compose logs consumer) каждые 5 секунд появляется текст, полученный от producer.
- Изменение содержимого data.txt на хосте приводит к отражению изменений в логах обоих сервисов.
- Ученик может показать файл docker-compose.yml и объяснить каждую секцию.
#### Примерный порядок работы
1. Клонировать репозиторий.
2. Отредактировать файл data.txt любым текстом (например, Hello from host).
4. Дополнить docker-compose.yml в соответствии с требованиями.
5. Запустить стек: docker-compose up.
6. Проверить статус: docker-compose ps.
7. Посмотреть логи: docker-compose logs -f.
8. Изменить содержимое data.txt, наблюдать обновление логов.
9. Остановить стек: docker-compose down.
10. Выполнить задания со звёздочкой (по желанию).

#### Шпаргалка по командам Docker Compose
| Команда | Описание                                                     |
|---------|--------------------------------------------------------------|
| `docker-compose up` | Запустить все сервисы                                        |
| `docker-compose down` | Остановить и удалить контейнеры (сети, volume остаются)      |
| `docker-compose down -v` | Остановить и удалить контейнеры вместе с volume              |
| `docker-compose ps` | Показать статус сервисов                                     |
| `docker-compose logs -f` | Показать логи всех сервисов (флаг -f следит за обновлениями) |
| `docker-compose logs producer` | Показать логи только producer                                |
| `docker-compose restart consumer` | Перезапустить только сервис consumer                         |
| `docker-compose exec consumer sh` | Зайти внутрь контейнера consumer                             |

#### Задания со звёздочкой
#### ⭐ 1 звезда: healthcheck и условная зависимость
- Добавить в сервис producer блок healthcheck, который проверяет эндпоинт /health (например, curl -f http://localhost:5000/health).
- В сервисе consumer изменить depends_on на условие service_healthy, чтобы consumer запускался только после того, как producer будет полностью готов.
- Проверить,
что consumer не запускается раньше, чем producer пройдёт healthcheck (с
учётом 5-секундной паузы при старте producer).
#### ⭐⭐ 2 звезды: комбинированный вывод из двух источников
- Изменить producer так, чтобы он выводил в консоль не только содержимое файла, но и значение переменной GREETING (например, в формате File: <content>, Env: <value>).
- Убедиться, что consumer получает от producer /data именно эту комбинированную строку.
- Продемонстрировать, что изменение data.txt и/или .env (и перезапуск) приводит к обновлению вывода.
- Файл .env должен находиться рядом с docker-compose.yml и содержать как минимум GREETING=Hello from env.
#### ⭐⭐⭐ 3 звезды: кастомная сеть и изоляция
- Создать в docker-compose.yml отдельную сеть (например, app-network).
- Поместить оба сервиса в эту сеть.
- Убрать у producer публикацию порта (ports), чтобы он был доступен только внутри сети.
- Убедиться, что из хоста больше нельзя обратиться к localhost:5000, но consumer по-прежнему получает данные.
#### Что показать при сдаче
Нужно показать преподавателю:
- Файл docker-compose.yml.
- Файл .env (без паролей в открытом виде).
- Файл data.txt с произвольным содержимым.
- Вывод docker-compose ps (оба контейнера в статусе Up).
- Логи producer (docker-compose logs producer) – видно паузу при старте, затем периодический вывод.
- Логи consumer (docker-compose logs consumer) – видно, как он получает данные.
- Демонстрация изменения data.txt и наблюдение за логами.
- Если есть звёздочки – соответствующие настройки и их демонстрация (healthcheck, условная зависимость, кастомная сеть).
